Objectives

What this design has to deliver

  • Run dbt unit tests and full-pipeline regression on a nightly schedule.
  • Execute inside a resettable GCP environment — no warm infrastructure to maintain or pay for.
  • Use a persistent bucket of CSV test inputs as the durable source of truth.
  • Drive everything from Python, triggered by a Harness pipeline.
  • Produce a report Harness auto-generates (no manual interpretation).
  • Notify the team in Microsoft Teams on completion, with detail on what drifted.
Nightly Regression PipelineObjectives
Design principles
  • Persistent inputs, disposable compute. Fixtures live in GCS; BigQuery state is recreated each run and thrown away.
  • One orchestrator, one report surface. Python (pytest) owns invocation, diffing, and reporting.
  • dbt does dbt. We wrap it, we don't reimplement it.
  • Determinism first. Frozen time and stable ordering, or reference outputs are worthless.
  • Granular failures. One drifted model = one failed test case, not a red wall.

Architecture overview

Four moving parts

GCS bucket

Versioned input CSVs (source stand-ins) and reference-output CSVs. Durable, auditable source of truth.

BigQuery dataset

Ephemeral, recreated each night. Cattle, not a pet.

Python / pytest

Loads fixtures, invokes dbt, runs regression diffs, emits JUnit, posts to Teams.

Harness

Schedules the run, ingests JUnit for the Tests tab, holds secrets and the GCP connection.

Flow:  GCS fixtures → BQ ephemeral dataset → dbt build → regression diff → JUnit → Harness report + Teams
Nightly Regression PipelineArchitecture
Environment model

Namespaced datasets from the outset, so the suite serves CI and nightly

  • A parameterised target dataset — test_regression_nightly for the scheduled run, test_regression_pr_<id> for per-PR CI — set via a Harness var and a generate_schema_name override.
  • Each dataset is truncated and reloaded (or created fresh) per run (bq mk --force); dbt auto-names and auto-expires its own temp tables within it, so parallel runs never collide.
  • GCS bucket holds inputs + reference outputs with object versioning on, so a re-approved reference output is auditable.
  • Inputs are version-controlled alongside (or generated from) the dbt project, so fixtures and the models that consume them move together.

Two test layers, two jobs

These are not the same test and shouldn't be conflated

dbt unit tests dbt 1.8+

  • Mock a single model's inputs, assert on its output rows.
  • Don't touch the bucket or real sources. Fast.
  • Run on BigQuery in CI (Harness) — the same adapter as prod, so no SQL dialect shimming.

Regression

  • Full-pipeline run against the bucket inputs, output diffed against reference outputs.
  • Catches type/coercion and cross-model interaction bugs.
  • Runs against the ephemeral BigQuery dataset for dialect/type fidelity.

pytest as the single orchestrator

Why pytest owns the whole thing

  • It is the single report producer — Harness only needs JUnit XML on disk.
  • It naturally turns each regression target and each dbt node into a discrete test case.
  • It keeps the orchestration in one language, matching the team's Python stack.
  • Failure messages land directly in the JUnit case, so Harness shows why, not just that.
dbt is invoked from inside pytest; its native test results are converted into the same report format.

Driving dbt from Python

Invoke programmatically (not a subprocess) to get the structured result object back:

dbt build runs unit tests, data tests, and models in dependency order in one pass. Per-node outcomes are written to target/run_results.json.
from dbt.cli.main import dbtRunner

def run_dbt_build(target="ci"):
    res = dbtRunner().invoke(["build", "--target", target])
    # don't raise on failure yet — we still want the report + Teams message
    return res

Loading fixtures from GCS

Load the CSV inputs into native typed source tables (not external tables) so the run behaves like prod and regression can catch type/coercion bugs:

from google.cloud import bigquery

client = bigquery.Client()

def load(table, uri):
    job = client.load_table_from_uri(
        uri,                                  # gs://.../inputs/trades.csv
        f"test_regression.{table}",
        job_config=bigquery.LoadJobConfig(
            source_format=bigquery.SourceFormat.CSV,
            skip_leading_rows=1,
            write_disposition="WRITE_TRUNCATE",
            schema=SCHEMAS[table],            # explicit schema, no autodetect drift
        ),
    )
    job.result()
Explicit schemas beat autodetect here — they keep the fixture contract stable night to night.

Regression diffs as pytest cases

One reference output per parametrized case, so a single drifted model is one failed case:

import pandas as pd, pytest
from google.cloud import bigquery

REFERENCE_OUTPUTS = ["fct_trades", "fct_orders", "fct_rfqs"]
client = bigquery.Client()

@pytest.mark.parametrize("model", REFERENCE_OUTPUTS)
def test_regression(model):
    expected = pd.read_csv(f"gs://.../reference/{model}.csv")
    actual = client.query(f"SELECT * FROM test_regression.{model}").to_dataframe()

    key = sorted(expected.columns)
    e = expected[key].sort_values(key).reset_index(drop=True)
    a = actual[key].sort_values(key).reset_index(drop=True)

    pd.testing.assert_frame_equal(e, a, check_dtype=False)
Diffing in Python keeps everything in one language. For matched / only-in-A / only-in-B detail, swap assert_frame_equal for a key-based set difference.

dbt results into the report

A small converter turns dbt's node results into JUnit so they share the Harness Tests tab. Written inline rather than depending on dbt-junitxml, which lags dbt releases:

import json
from junit_xml import TestSuite, TestCase

results = json.load(open("target/run_results.json"))["results"]
cases = []
for r in results:
    tc = TestCase(r["unique_id"], elapsed_sec=r["execution_time"])
    if r["status"] in ("error", "fail"):
        tc.add_failure_info(message=r.get("message", r["status"]))
    elif r["status"] == "skipped":
        tc.add_skipped_info(message="skipped")
    cases.append(tc)

with open("reports/dbt-junit.xml", "w") as f:
    TestSuite.to_file(f, [TestSuite("dbt", cases)])

Determinism: the thing that bites

Non-deterministic output makes reference outputs worthless and produces phantom failures. Three controls:

01

Frozen as-of time

Inject a fixed timestamp via a dbt var; models read it instead of current_timestamp() / current_date(). Critical for the bi-temporal models.

02

Stable ordering

Compare as sets — sort on key columns before diffing, never rely on row sequence.

03

Deterministic surrogate keys

Hashing must be reproducible across runs.

How the auto-generated report works

  • Point it at a glob: reports/*-junit.xml.
  • Both the pytest output and the converted dbt output merge into one Tests tab — per-case, with status and duration.
  • No extra plugins or config — JUnit on disk is the entire contract.

Teams notification · code

Python posts a summary richer than a generic pass/fail — it names what drifted:

import os, requests

def notify(summary):
    card = {
        "type": "message",
        "attachments": [{
            "contentType": "application/vnd.microsoft.card.adaptive",
            "content": {
                "type": "AdaptiveCard", "version": "1.4",
                "body": [
                    {"type": "TextBlock", "size": "Large", "weight": "Bolder",
                     "text": f"Nightly regression: {summary['status']}"},
                    {"type": "TextBlock", "wrap": True,
                     "text": f"dbt {summary['dbt_pass']}/{summary['dbt_total']} · "
                             f"regression {summary['reg_pass']}/{summary['reg_total']}"},
                    {"type": "TextBlock", "wrap": True,
                     "text": "Drift: " + (", ".join(summary['failures']) or "none")},
                ],
            },
        }],
    }
    requests.post(os.environ["TEAMS_WORKFLOW_URL"], json=card, timeout=10).raise_for_status()

Wiring & failure strategy

The details that decide whether you actually get told things broke

  • Set teardown and notify_teams.py to always execute in the Harness failure strategy — a failed assertion must not skip the Teams message.
  • Don't let pytest's non-zero exit abort the pipeline before the report and notify steps run. Either convert the report in the same pytest invocation (a conftest hook) or set the step to continue and gate afterward.
  • Keep teardown guaranteed so a failed run never leaks a dataset or cost.

Technical aspects to prove out ASAP

The spikes that carry the most risk — validate these before building out:

  • Dataset namespacing & CI reuse. Confirm the parameterised target dataset + generate_schema_name override cleanly isolates nightly vs per-PR runs, and that dbt's auto-expiring temp tables don't leak.
  • Frozen as-of propagation. Confirm the dbt var threads through the bi-temporal models and yields byte-stable reference outputs across two consecutive runs.
  • dbtRunner result shape. Verify structured results / run_results.json parse cleanly on our pinned dbt version.
  • Harness multi-JUnit ingestion. Confirm the glob renders both files per-case in the Tests tab.
  • Power Automate Workflows webhook. Provision it, confirm ownership, payload shape, Flow-bot identity is acceptable, and private-channel support if needed.
  • WIF end-to-end. Harness → GCP with the exact BQ + GCS permissions.
  • Nightly window. Full reload + build + diff completes within budget; check gcsfs (gs:// reads) and BigQuery Storage API (to_dataframe) deps.
  • Reference-output approval workflow. How a reference output gets re-approved and versioned, and the frame-vs-row-level diff decision for wide tables.
