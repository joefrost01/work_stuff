Confluence Space Build Playbook

A manual, click-by-click guide to building the lab space on Confluence Cloud. Work through the phases in order — each ends with a checkpoint so you know it's safe to continue. Budget roughly half a day, spread over a week if you prefer.

Phase 0

Prerequisites & decisions

~20 min

Do these before touching Confluence. Everything later depends on them.

  1. Confirm you can create spaces: open Confluence, click Spaces in the top bar — you need a Create a space option. If missing, ask your site admin for space-creation permission.
  2. Confirm you can edit Jira issues in your lab's Jira project(s) and that the Labels field is visible when you create an issue. If it isn't, you'll need a Jira project admin in Phase 11 — line one up now.
  3. Write down your product list from the design deck: every ODP (by source system — Summit-ALXXXXXX, ReD-ALXXXXXX, Fenics-ALXXXXXX, etc.), every FDP (by canonical entity — Event.Transaction.Trade, Event.Interaction.Orders, etc.), and the two CDP areas (Surveillance, Trading Controls). You'll type these many times; a fixed list prevents naming drift.
  4. Decide the space name and key now: name Markets Lab (or your preferred name), key MLAB. The key appears in every URL and cannot be changed later.
Checkpoint 0You have space-create rights, a Jira contact, a written product list, and a chosen space name + key.
Phase 1

Create the space

~10 min
  1. Top bar → Spaces → Create a space. If offered a template gallery, choose Blank space — we are building our own structure, a template will fight you.
  2. Enter the name and key you decided in Phase 0. Set access to your team for now (you'll open it up in Phase 9).
  3. Open the new space's home page → Edit → rename the title to Lab Home → Publish. Leave the body empty; you'll fill it in Phase 10, once everything it links to exists.
Checkpoint 1The space exists, and its homepage is titled "Lab Home".
Phase 2

Lock the label taxonomy

~20 min

Labels are the machinery of the whole design — get them right before creating content pages.

Test dotted labels first. Your naming pattern uses dots (fdp-event.transaction.trade). Jira accepts dots in labels; Confluence Cloud rejects some punctuation. Before committing: open any page, press L, and try adding fdp-event.transaction.trade. If Confluence rejects or mangles it, switch the whole taxonomy to dashes (fdp-event-transaction-trade) — in BOTH systems. The strings must match exactly or every roll-up silently breaks.
  1. Run the dot test described above and note the result.
  2. Write the master taxonomy list. Start from this and extend with your product list from Phase 0: 
    # Concern labels (bare nouns — used on pages AND Jira tickets)
adr   recon   release   governance   runbook   service-intro   control-test   incident

# Component labels (one per entry in the Component Catalogue)
comp-ingestion   comp-cicd   comp-trading-hub   comp-alerting   comp-controls-portal   comp-ai-services

# Product labels (one per product — examples, complete from your list)
odp-summit-alxxxxxx   odp-red-alxxxxxx   odp-fenics-alxxxxxx
fdp-event-transaction-trade   fdp-event-transaction-order
cdp-surveillance   cdp-trading-controls
  3. Rules to enforce forever: all lowercase · no spaces · one label per product/component, reused verbatim everywhere · never invent a label mid-task — extend the master list first.
Checkpoint 2You have a final written label list, dot-or-dash decided, covering every product, component and concern.
Phase 3

Top-level skeleton

~15 min

Create the six zone pages as children of Lab Home. In the left sidebar, hover Lab Home → click the + that appears → a new child page opens. Title it, publish with an empty body, repeat.

  1. Create child page How the Lab Works → Publish.
  2. Create child page Data Products → Publish.
  3. Create child page Releases & Operations → Publish.
  4. Create child page Teams → Publish.
  5. Create child page Governance & Assurance → Publish.
  6. Create child page _Master Copies → Publish.

    The underscore keeps this utility branch sorted last — it's scaffolding, not content.

  7. In the sidebar, drag the six pages into exactly this order (top to bottom): How the Lab Works · Data Products · Releases & Operations · Teams · Governance & Assurance · _Master Copies.
Always create pages from the sidebar + on the intended parent. Pages created from the global Create button land at space root and you'll have to move them. If a page lands in the wrong place: ••• menu → Move, or drag it in the sidebar.
Checkpoint 3Lab Home has exactly six children, in the order above.
Phase 4

Master copies — your "templates"

~45 min

Confluence templates can only create single pages — they cannot create a page tree. Since every data product needs eight child pages, we use a master-copy branch instead: build the tree once under _Master Copies, then copy-with-children for each real product. This is the single most important trick in this playbook.

4a · The data product master

  1. Under _Master Copies, create child page ZZ Product Name.
  2. In its body, add a Page Properties macro: type /page properties → select it → inside the macro insert a two-column table with these exact rows (left column bold): 
    Owner        | (name)
Squad        | (squad)
Type         | ODP | FDP | CDP   ← delete two, keep one, when copying
Status       | Draft
Criticality  | TBC

    These rows power the auto-generated Product Registry in Phase 6 — the row names must be identical on every product page.

  3. Publish, then create these eight child pages under it, each published with a one-line description of what belongs there: 
    Overview & Ownership
Data Model & Contracts
Governance & Controls
Lineage
DQ & Reconciliations        ← rules, links to recon pages, CI details
Operations
Decisions
Work in Flight
  4. On Decisions: add a Content by Label macro (type /content by label), configured with label adr. You'll add the product's own label to this filter on each copy.
  5. On Work in Flight: add a Jira macro (type /jira) with placeholder JQL labels = REPLACE-ME ORDER BY updated DESC.

4b · Single-page templates

These are genuinely single pages, so real Confluence templates work: go to Space settings → Templates → Create a template (wording varies slightly by version) and create:

  1. Template ADR — headings: Context · Decision · Status · Consequences · Products affected. Reminder line at top: "Add labels: adr + every product/component label this touches."
  2. Template Release Note — headings: Summary · Contents · Products & components affected · Evidence / gates. Reminder: "Add labels: release + affected product/component labels."
  3. Template Reconciliation — headings: Purpose · Scope · Owner · Schedule · Break process · Evidence location. Reminder: "Add labels: recon + product/component labels."
  4. Template Team Page — headings: Mission · Members · Rituals · Roadmap · Jira board (link) · What we own (links only).
If you can't find space templates in your plan/UI, skip them: create four more master pages under _Master Copies instead (ZZ ADR, ZZ Release Note, ZZ Reconciliation, ZZ Team Page) and copy them when needed. Identical outcome.
Checkpoint 4_Master Copies contains "ZZ Product Name" with its 8 children; the four single-page templates exist (either as space templates or ZZ masters).
Phase 5

How the Lab Works

~40 min

All pages in this phase are created under How the Lab Works using the sidebar +.

  1. Create Architecture, and under it: Reference Architecture & Principles, Integration Patterns, System Component Catalogue, ADR Log.
  2. Under System Component Catalogue, create one page per component (one per comp-* label from Phase 2). On each: press L → add its own comp-* label → in the body add a Jira macro with JQL labels = comp-xxx ORDER BY updated DESC (matching that page's label).
  3. On ADR Log: add a Content by Label macro filtered to label adr, restricted to this space. New ADRs are created under this page from the ADR template — the log assembles itself.
  4. Create Infrastructure & Platform, and under it: Paved Road — CI/CD & Environments, GCP Estate, Ingestion Frameworks, Shared Services Catalogue, Access & Onboarding.
  5. Create Lab Engagement, and under it: How to Engage the Lab, Intake to Delivery Lifecycle, Prioritisation & Planning, RACI & Ways of Working.
  6. Create Standards & Templates, and under it: Label & Metadata Taxonomy ← paste your Phase 2 master list here; this page is now the single source of truth — plus Data Product Documentation Standard, Data Modelling Standards, DQ & Reconciliation Standards, Release & Change Standards, Service Readiness Checklist.
Checkpoint 5Four branches exist under How the Lab Works; the taxonomy page holds the master label list; every component page has its label + Jira macro.
Phase 6

Data Products

~60 min
  1. Under Data Products, create: Product Registry, Lineage Map, ODPs, FDPs, CDPs (in that order).
  2. Now stamp out products. For each product on your Phase 0 list: open ZZ Product Name → ••• menu → Copy → tick Include child pages → set destination to ODPs, FDPs or CDPs as appropriate → copy.
  3. On each copy, immediately do all four of these before moving to the next product — half-finished copies are how mistakes creep in:

    ① Rename the page to the product (e.g. Summit/Murex ODP, Trade FDP, Surveillance CDP). ② Press L, add its product label (e.g. fdp-event.transaction.trade). ③ Fix the Page Properties table: set Type to just ODP/FDP/CDP, fill Owner and Squad. ④ Open its Work in Flight child and replace REPLACE-ME in the JQL with the product label; open Decisions and add the product label to the Content-by-Label filter.

  4. On Product Registry: add a Page Properties Report macro (type /page properties report). Configure: Label = your product labels won't all fit, so instead add label data-product — then go back and press L on every product page to add data-product alongside its own label. The report now lists every product with Owner / Squad / Type / Status columns, automatically.
  5. On Lineage Map: insert a simple table or diagram showing ODP → FDP → CDP flows, linking each name to its page. (A table of "consumes → consumed by" rows is fine to start; replace with a diagram later.)
Copy direction matters. Always copy FROM the ZZ master, never from another product page — earlier products accumulate real content you don't want duplicated into new ones.
Checkpoint 6Every product from your list exists with 8 children, its own label + data-product, correct Page Properties, and a working Jira macro. The Product Registry lists them all.
Phase 7

Releases & Operations

~30 min
  1. Under Releases & Operations, create: Releases, Reconciliations, Incidents & Problems, Runbooks & Support.
  2. Under Releases: create Release Calendar, Release Process & Gates, and Release Notes. On Release Notes, add a Content by Label macro filtered to release, this space. Individual release pages are created under it from the Release Note template.
  3. Under Reconciliations: create Recon Register with a Content by Label macro filtered to recon. Individual recs are created under Reconciliations from the Reconciliation template, each labelled recon + its product/component labels.
  4. Under Incidents & Problems: create Incident Log (a simple table to start) and Post-Incident Reviews.
  5. Under Runbooks & Support: create Support Model & Rotas and Runbook Index — the index is a Content by Label macro on runbook. Runbooks themselves live under each product's Operations page, labelled runbook + product label.
Checkpoint 7The three registers (release notes, recons, runbooks) are label-driven macros, not hand-maintained lists.
Phase 8

Teams

~25 min
  1. Under Teams, create one page per team from the org slide using the Team Page template: Lab Leadership, CICD & DevOps, Data Engineering, Integration & APIs, QE / Automation / XRA, AI & Advanced Analytics, Shared Data Product Squad, Surveillance Squad, Trading Controls Squad.
  2. On each team page, fill "What we own" with links to its products and components — select text → Ctrl/Cmd+K → search the page name. No product documentation is ever written here.
Checkpoint 8Nine team pages, each linking out to owned products/components, hosting nothing.
Phase 9

Governance & Assurance

~35 min
  1. Under Governance & Assurance, create: Start Here — Audit, Risk & Security, Controls Matrix, Governance Records, Recon & Control Test Results, Security, Audit Workspaces.
  2. Governance Records: Content by Label macro on governance. Then press L on every product's Governance & Controls child page and add the governance label — that single action populates this whole view.
  3. Recon & Control Test Results: two Content by Label macros — one on recon, one on control-test.
  4. Controls Matrix: a table with columns Risk · Control · Evidence, where Evidence cells are links into product Governance pages and recon pages. This is the one assurance page you maintain by hand.
  5. Under Security: create Access Model Summary and Assessment Evidence Index (Content by Label on a security-evidence label if you want it self-maintaining — add the label to your taxonomy page first).
  6. Grant access: Space settings → Space permissions (or Access) → give your audit, risk and security groups View permission. They should not need Edit.
Checkpoint 9An auditor with view access can reach any product's governance evidence in two clicks from Start Here, and the roll-ups populate themselves.
Phase 10

Lab Home front doors

~20 min
  1. Edit Lab Home. Add a heading Start here and a five-row layout (or simple list) — one entry per audience, each a single bold link with one line of description: 
    Engineers            → How the Lab Works
Product & business   → Data Products
Audit & risk         → Governance & Assurance › Start Here
Security             → Governance & Assurance › Security
Ops & support        → Releases & Operations
  2. Below it, add a short "What is this space?" paragraph and a link to the Label & Metadata Taxonomy page.
  3. Optionally embed your two slide decks (org structure + space design) as attachments or links for context.
Checkpoint 10Every audience reaches its zone in one click from Lab Home.
Phase 11

Jira wiring

~30 min + admin help
  1. With your Jira project admin: ensure the Labels field appears on the create screen of every issue type the lab uses (Jira admin → Screens). If people must open an issue after creation to label it, labelling will not happen.
  2. Paste the label taxonomy into the Jira project's description or a pinned page, linking back to the Confluence taxonomy page as the source of truth.
  3. Spot-check the round trip: create a test Jira ticket labelled with one product label (e.g. cdp-surveillance) → open that product's Work in Flight page in Confluence → confirm the ticket appears in the Jira macro. Delete the test ticket after.
  4. Optional but recommended: create saved Jira filters per product label (labels = cdp-surveillance ORDER BY updated DESC) so boards and dashboards can reuse them.
If the Jira macro shows "no issues found" when you know tickets exist: 9 times out of 10 the label string differs (dot vs dash, typo, case). Compare the page label and the ticket label character by character.
Checkpoint 11A labelled test ticket appeared on the right product page with zero manual curation.
Phase 12

Final verification — the pilot

~half a day, spread out

Don't announce the space yet. Prove it with one product first.

  1. Pick one FDP and document it end-to-end: fill all eight child pages with real content, including data model, governance, and at least one real reconciliation page linked from DQ & Reconciliations (with rules, recon links and CI details).
  2. Create one real ADR from the template, labelled with that FDP — confirm it appears on both the global ADR Log and the product's Decisions page.
  3. Create one release note labelled with the FDP — confirm it appears in the Release Notes register.
  4. Ask one engineer, one PO and (if possible) one risk/audit colleague to find three things each, unaided. Watch where they stumble; fix navigation, not people.
  5. Walk the full checklist: ☐ six zones in order ☐ registry lists all products ☐ all three registers self-populate ☐ Jira round trip works ☐ assurance has view access ☐ taxonomy page is linked from Lab Home.
  6. Announce the space, present the design deck, and state the three habits: label at creation · templates are law · every page has an owner.
DoneOne product fully documented, every automatic view proven live, and three real users navigated it unaided. Now scale out product by product.
