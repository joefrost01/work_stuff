Part One

Data quality: dbt tests vs DataPlex

Both can catch bad data. They sit in different layers, operate on different signals, and answer to different teams.

Option A · In-warehouse, code-first

dbt data quality checks

What it is

Assertions defined in YAML alongside the model — not_null, unique, accepted_values, relationships, plus singular SQL tests and dbt-expectations for richer rules.

Where it runs

As SQL inside the warehouse (BigQuery), on the same run that builds the model. Failures break the build.

Merits

  • Lives with the model — tests version-controlled in the same PR as the SQL they validate
  • Cheap and fast — runs as part of the existing dbt invocation; no extra infrastructure
  • Blocks bad data at build time — a failing test can fail the pipeline before downstream models materialise
  • Engineer-native — diffs in Git, reviewed in PRs, surfaced in CI
  • Composable — packages like dbt-expectations and dbt-utils cover the common ground
01 · Data Qualitydbt
Option B · Platform-managed, catalog-driven

Google DataPlex data quality

What it is

A managed DQ service that runs rule-based scans across BigQuery tables and lakes, with results published into DataPlex/Data Catalog metadata, dashboards, and Cloud Logging.

Where it runs

As scheduled scans on the Google side, decoupled from the build. Rules are defined in YAML or the console; results surface in the catalog and Looker.

Merits

  • Cross-asset visibility — one pane of glass for DQ across BQ, GCS, and other registered assets
  • Catalog integration — DQ scores attach to the asset; stewards and consumers see them in context
  • Operates on landed data — catches issues in tables we didn't build (raw ingest, vendor drops)
  • Lineage & profiling — pairs naturally with column-level profiling and discovery
  • Steward-friendly — rules can be authored without touching the pipeline repo

Part Two

Testing the pipeline itself, not just the data

DQ tests assert on real data after the fact. Unit tests assert on transformation logic against fixed inputs — before anything runs in production.

Proposal

dbt unit tests, executed in Harness

What changes

Today, dbt data tests run after the model materialises. dbt's unit testing framework (1.8+) lets us assert on a model's SQL with mocked inputs and expected outputs — no warehouse data required.

Where it runs

As a stage in our Harness pipeline, on every PR and pre-merge. Pure SQL evaluation against fixtures — fast, deterministic, no production data touched.

Why this matters

  • Catches logic regressions in CASE statements, joins, and window functions that data tests miss
  • Deterministic — fixtures don't drift; the same input always produces the same expected output
  • Fast feedback — runs in seconds on a PR, no warehouse spend, no scheduled scan latency
  • Documents intent — the fixture is the spec; reviewers can read it and understand the rule
  • Plays nicely with Harness — same image, same dbt project, just an extra stage

What it looks like

A unit test, and where it sits in Harness

dbt unit test (YAML)

unit_tests:
  - name: fx_trade_pnl_handles_fx_reset
    model: fct_trade_pnl
    given:
      - input: ref('stg_trades')
        rows:
          - {trade_id: 1, ccy: GBP, notional: 100, fx_rate: 1.00}
          - {trade_id: 2, ccy: USD, notional: 100, fx_rate: 1.25}
      - input: ref('dim_fx_reset')
        rows:
          - {ccy: USD, reset_rate: 1.30}
    expect:
      rows:
        - {trade_id: 1, pnl_gbp: 100}
        - {trade_id: 2, pnl_gbp: 130}

03 · ConsumptionRequirement
Proposal

Python extracts using Polars

Shape of the solution

  • A small Python package, one extract definition per JSON output
  • Each definition: a SQL query (or dbt mart reference), a Polars transform, a JSON writer, a destination
  • Orchestrated daily — same Harness pipeline patterns we use elsewhere
  • Schema-validated on the way out (Pydantic / JSON Schema)
  • Idempotent, with a manifest of what was sent and when

Why Polars

  • Fast columnar engine, lazy execution, predictable memory
  • Native reads from BigQuery / Parquet, native JSON writing
  • Tiny dependency surface vs. Spark / Beam / a service framework
import polars as pl
from gcp import bq, gcs

def build_positions_extract(run_date):
    df = (
        bq.read("gold.positions_eod")
          .filter(pl.col("as_of") == run_date)
          .select(["book", "instrument_id", "qty", "pnl_gbp"])
          .sort("book", "instrument_id")
    )
    validate(df, schema=POSITIONS_SCHEMA)
    payload = df.write_json(row_oriented=True)
    gcs.write(
        bucket="tradinghub-outbound",
        key=f"positions/{run_date}.json",
        body=payload,
    )
    manifest.record("positions", run_date, df.height)

What we're choosing not to build

And why not

Considered

A REST API

Trading Hub doesn't pull — they consume files. Standing up an API gives us auth, scaling, and runtime concerns for zero functional benefit.

Considered

A semantic / serving layer

Useful when many consumers ask varied questions. We have one consumer asking the same questions every day.

Considered

Spark / Beam job

Volumes don't justify a distributed engine. Polars on a single worker is faster end-to-end and cheaper to operate.

Right-sized wins: a few hundred lines of Python, version-controlled with the rest of the platform, deployed via the same Harness templates, observable with the same tooling.
03 · ConsumptionTrade-offs
In summary

Three proposals, one principle

DQ

dbt-first, DataPlex on top

  • dbt tests for everything we build
  • DataPlex for landed / shared / steward-facing
  • Both for regulated gold-tier assets
Testing

dbt unit tests in Harness

  • Catch logic regressions pre-merge
  • Deterministic fixtures, no warehouse cost
  • One extra stage in the existing pipeline
Consumption

Polars-based daily extracts

  • Right-sized for a single-consumer, file-based brief
  • Lean, observable, Git-versioned
  • No API or platform overhead to justify
Common thread: keep the tools close to the code, keep the surface small, and only reach for platform-scale solutions when the platform-scale problem actually shows up.
