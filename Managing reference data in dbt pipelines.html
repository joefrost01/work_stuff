PROBLEM · BASELINE

Two needs that pull in opposite directions

Reference data has to serve two audiences with very different expectations — and the obvious tools each only solve half the problem.

Need 1

Day-to-day updates by ops

Traders move desks, exchanges add holidays, venues retire. These edits are routine and ops-owned.

  • Must not require a git PR or an engineer.
  • Must be auditable — who changed what, when.
  • Must be reversible — history queryable in SQL.

dbt seed fails this — every edit is a deploy event.

Need 2

Standing up new environments

A fresh dev / UAT / DR environment must come up with all reference data populated — the pipeline can't run without it.

  • No human-in-the-loop on day one.
  • Defaults must ship with the code.
  • Re-baseline must be one command, not many.

Pure ad-hoc uploads fail this — nothing seeds a new env.

What we want. One mechanism that satisfies both — ops can edit at runtime, and deploys can provision a new environment from scratch — without two ingestion patterns to maintain.

SHAPE · ONE PIPELINE

One loader. Two doors. Same code path.

Every reference table is loaded by the same Python module, regardless of who triggered it. The only thing that differs is where the CSV came from.

Door 1deploy-time

defaults/

Repo holds a baseline CSV for every static_ref_* table. Harness mirrors them to gs://ref/defaults/<env>/ on every deploy.

  • Always current with the shipped code.
  • Loaded on first run in a new environment.
  • Schema additions can ship a starter CSV alongside.
Door 2ops-driven

inbox/

Ops drop a CSV into gs://ref/inbox/ when they need to change something. Filename matches the target table.

  • No git PR, no engineer in the loop.
  • Beats the default if both are present.
  • Triggered manually — promotion is a decision.
The split is by intent, not by data tier. Where a CSV came from tells you what the loader should do; what's in it doesn't.

CONVENTION · NAMING

One name, three artifacts

Every ref table is a static_ref_* in BigQuery and is fully described by three same-named artifacts. The convention is the registry — no per-table code, no central config.

trader_id_to_name is illustrative — the final prefix, paths, and naming style will be agreed at implementation; the structure stays the same.

Per table — three files in the repo

data-pipeline/                  ← this repo
  ref_defaults/
    trader_id_to_name.csv          ← baseline data
    trader_id_to_name.schema.yml   ← semantic rules

platform-infra/                 ← separate Terraform repo
  bigquery/ref/
    static_ref_trader_id_to_name.tf  ← structural schema

Adding a ref table is a coordinated change — a Terraform PR for the table, a pipeline PR for the CSV + schema yaml. CI lints them against each other.

SCHEMA · TWO LAYERS

Terraform enforces structure. YAML enforces semantics.

BigQuery enforces column types and NOT NULL. It does not enforce UNIQUE, CHECK, or row counts. We put each rule where it can actually be enforced — and CI lints the two for drift.

Layer 1Terraformstatic_ref_trader_id_to_name.tf
resource "google_bigquery_table" "static_ref_trader_id_to_name" {
  dataset_id = "ref"
  table_id   = "static_ref_trader_id_to_name"
  schema = jsonencode([
    { name = "trader_id",  type = "STRING",    mode = "REQUIRED" },
    { name = "name",       type = "STRING",    mode = "REQUIRED" },
    { name = "valid_from", type = "TIMESTAMP", mode = "REQUIRED" },
    { name = "valid_to",   type = "TIMESTAMP", mode = "NULLABLE" },
  ])
  time_partitioning { type = "DAY", field = "valid_from" }
  clustering = ["trader_id"]
}

Owns: column types, NOT NULL, partitioning, clustering. Enforced by BigQuery on insert.

Layer 2YAMLtrader_id_to_name.schema.yml
# Semantic rules — things BQ can't enforce itself.
columns:
  trader_id:
    csv_name: "Trader ID"                # optional; case-insensitive by default
    unique_active: true                  # unique among valid_to IS NULL
    regex: "^[A-Z]{3}[0-9]{4}$"
  name:
    max_length: 80

row_count: { min: 50, max: 2000 }

# Cross-column rules
rules:
  - name: "no duplicate active names"
    sql: "countif(valid_to is null) = count(distinct name)"

Owns: unique, regex, row counts, CSV remapping, domain rules. Enforced by the loader before insert. Extra CSV columns are warned and skipped.

CI lints both for drift. A small script reads each .tf and matching .schema.yml, asserts they agree on column names and that every yaml-referenced column is REQUIRED in Terraform unless explicitly nullable. Build fails on mismatch.

CODE · LOADER

One Python module, every table

Read with Polars, validate against the schema yaml, then close-out and insert in one BQ transaction. Defence in depth — Polars catches bad data first; BQ's REQUIRED columns are the transactional backstop. 

import polars as pl, yaml, logging
from google.cloud import bigquery
from datetime import datetime, timezone

def load(csv_path: str, schema_path: str, table: str) -> None:
    schema = yaml.safe_load(open(schema_path))
    df = pl.read_csv(csv_path)                                  # Polars reads gs:// directly

    # 1 · Resolve CSV columns → table columns (case-insensitive, optional csv_name)
    csv = {c.lower(): c for c in df.columns}
    mapping = {t: r.get("csv_name", t).lower() for t, r in schema["columns"].items()}
    if missing := [t for t, c in mapping.items() if c not in csv]: raise ValueError(f"CSV missing: {missing}")
    if extras := set(csv) - set(mapping.values()): logging.warning(f"ignoring CSV columns: {sorted(extras)}")
    df = df.select([pl.col(csv[mapping[t]]).alias(t) for t in schema["columns"]])

    # 2 · Validate semantic rules (BQ enforces types + NOT NULL at insert)
    rc = schema["row_count"]
    assert rc["min"] <= df.height <= rc["max"], f"row count out of bounds: {df.height}"
    for col, r in schema["columns"].items():
        s = df[col]
        if r.get("unique_active"):     assert s.n_unique() == s.len(), f"{col} not unique"
        if rx := r.get("regex"):       assert s.str.contains(rx).all(), f"{col} regex fail"
        if m := r.get("max_length"):   assert s.str.len_chars().max() <= m, f"{col} too long"

    # 3 · Land the new snapshot in staging
    client, now = bigquery.Client(), datetime.now(timezone.utc)
    staging = f"{table}__staging"
    client.load_table_from_dataframe(
        df.with_columns(valid_from=pl.lit(now)).to_pandas(), staging,
        job_config=bigquery.LoadJobConfig(write_disposition="WRITE_TRUNCATE"),
    ).result()

    # 4 · Close-out + insert, atomically
    cols = list(schema["columns"]) + ["valid_from"]
    client.query(
        f"BEGIN TRANSACTION; "
        f"UPDATE `{table}` SET valid_to=@now WHERE valid_to IS NULL; "
        f"INSERT INTO `{table}` ({', '.join(cols + ['valid_to'])}) "
        f"SELECT {', '.join(cols)}, NULL FROM `{staging}`; "
        f"COMMIT TRANSACTION;",
        job_config=bigquery.QueryJobConfig(query_parameters=[
            bigquery.ScalarQueryParameter("now", "TIMESTAMP", now)]),
    ).result()

ORCHESTRATION · ONE DAG

One DAG. Two modes. Three callers.

A single Composer DAG wraps the loader. What it does on any given run is driven by one optional param — the GCS path of a file to load.

Mode Ano path param

env_check

List every static_ref_* table in BQ. For each one with no active rows: prefer inbox/<name>.csv, fall back to defaults/<env>/<name>.csv, load it. If neither exists, log loudly and continue.

  • Self-heals new environments.
  • Idempotent — populated tables are skipped.
  • Ops can pre-stage overrides in inbox/ before first run.
Mode Bpath = gs://...

Snapshot replace

Validate the file, close every currently-active row, insert the new snapshot — in one BQ transaction. Run regardless of whether the table is empty; this is the day-to-day update path.

  • Table name inferred from filename — no mapping config.
  • Schema yaml looked up from the deployed code, not from the CSV.
  • Archived to gs://ref/archive/<ts>/ on success.

Caller · Harness

Post-deploy

Mirrors ref_defaults/ to GCS, triggers the DAG in env_check mode. Self-provisions any missing tables.

Caller · Ops

On-demand

Drop CSV in inbox/, trigger the DAG with that path. Promotion is a deliberate two-step.

Caller · Schedule

Nightly safety net

Runs env_check on a cron — catches anything that drifted to empty (e.g. a TRUNCATE in dev).

TESTING · TWO LAYERS

Two independent layers of testing

The loader catches bad files before they land. Two further layers answer different questions about what's already in the warehouse. Time-based freshness doesn't apply — updates are ad-hoc.

Layer 1dbt

Source tests

Is the active snapshot well-formed every time dbt builds against it?

  • unique, not_null, regex — all scoped where valid_to is null.
  • Auto-generated from each *.schema.yml — single source of truth.
  • Runs with dbt build.
Answers: is it well-formed?
Layer 2Composer

Quality DAG

Are the values themselves correct, by domain rules, even if no upload has happened?

  • Runs daily, decoupled from uploads and from dbt runs.
  • Domain assertions — every active trader has a desk, every active venue has an MIC.
  • Logs to DynaTrace; failures propagate to ServiceNow tickets.
Answers: is it right?
The yaml feeds three places. The loader's runtime validation, dbt's source tests, and the quality DAG's domain checks all read from *.schema.yml — add a rule once, three layers pick it up.

OPERATING MODEL · THE LOOP

The testing loop

Two cadences, two responsibilities. Together they cover model correctness and reference-data quality — without coupling either to the upload schedule.

Cadence · hourly / daily

dbt build

Builds and tests the analytics models that use the ref tables, plus source tests on the active snapshot.

  • Owned by analytics eng.
  • Catches model correctness regressions.
  • Source tests are scoped to valid_to IS NULL.
  • Tied to the deploy SLA.

Cadence · daily

Quality DAG

Domain assertions on the current snapshot — runs whether or not anything was uploaded that day.

  • Owned by ops + data platform.
  • Catches content drift the schema can't see.
  • Runs whether or not dbt ran, whether or not ops uploaded.
  • Logs to DynaTrace; selective error propagation to ServiceNow.
Key principle. The deploy pipeline tests model correctness; reference data needs its own always-on quality loop. With ad-hoc updates there's no upload cadence to lean on — the quality DAG is the SLA.

Summary

One mechanism

  • One Python loader — Polars validate, BQ close+insert.
  • One DAG, two modes: env_check or targeted path.
  • Three callers — Harness, ops, schedule — same code path.

One convention

  • static_ref_* tables — the warehouse is the catalog.
  • Filename = table = schema yaml = Terraform resource.
  • Add a ref table: drop three files, deploy.

Two test layers

  • dbt source tests — active snapshot is well-formed.
  • Quality DAG — daily content correctness.
  • Both auto-generated from the same schema yaml.
The key principle. One mechanism serves both audiences — ops edit at runtime through inbox/; deploys provision new environments from defaults/ via Harness. Same loader, same validation, same SCD-2 history. No second ingestion pattern to maintain.
